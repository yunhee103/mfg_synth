# 4단계 분석 스프린트 가이드

대상: queries/의 쿼리 9본 (Q5는 ETL 품질 리포트로 대체 - 판단 3.6)
기간: 2일.

## 실행 방법

DBeaver에서 mfg_dw 접속 후 파일을 열어 블록 단위로 실행하거나:

    Get-Content queries/07_납기준수.sql -Raw |
      docker exec -i mfg_erp_source mariadb -uroot -pdevpass mfg_dw

## 권장 순서 (증상 -> 원인 사슬)

1. **Q7 납기 준수** - 이 회사의 "증상"이 여기 있다. 월별 추이에서
   꺾이는 달을 먼저 찾아라. 이후 모든 질문은 이 증상의 원인 후보를
   좁히는 과정이다.
2. **Q3 미출하** - 빠르게 끝나는 판정 하나. 탐색 워크시트의 "판별
   보류"를 여기서 닫는다.
3. **Q8 -> Q9 (설비 체인)** - Q7c가 가리킨 라인을 Q8에서 추적.
   변화 "시작 주"를 특정하는 것이 목표. Q9에서 불량과의 동조 확인.
4. **Q6 -> Q10 -> Q1 (자재 체인)** - 상습 지연 공급사와 시작 시점
   특정 -> 결품 자재/시기 -> 대기 작업지시와 겹침 확인.
5. **Q2, Q4 (데이터 품질)** - 대사 불일치의 분포 판정, 코드 유형 집계.

## 기록 양식 (Q마다, 리포트의 원료가 된다)

- 발견 한 줄: (예: "X라인의 지표가 N월 M주부터 K% 저하")
- 근거 쿼리/수치: (파일명 + 핵심 숫자 2~3개)
- 판단: (이것이 증상의 원인 사슬에 들어가는가, 별개 문제인가)
- 남는 의문: (확인 못 한 것 - 리포트의 "한계" 절 재료)

## 규칙

1. ground_truth.json은 열지 않는다. 분석이 전부 끝나고 리포트 초안이
   나온 뒤에만 채점용으로 연다. (Claude Code 병용 시 세션 지시문에
   "output/ground_truth.json 읽기 금지"를 명시할 것)
2. 플레이스홀더(__여기에_...__)는 앞 쿼리 결과로 본인이 특정해 넣는다.
   6b(공급사코드), 8b/8c(설비코드) 두 곳.
3. 쿼리 수정은 자유. 단 수정본은 파일에 남겨서 리포트의 근거로 쓴다.
4. 해석 검증이 필요하면 "결과 숫자 + 내 해석"을 함께 가져올 것.
   해석 없이 숫자만 가져오면 해석을 대신해주지 않는다.

## 참고

- 01번 1c는 BOM이 창고에 미수록이라 erp_source.bom을 교차 참조한다.
  이는 설계 리허설이 잡지 못한 갭으로, 리포트 "한계/개선" 절의 재료다.
- 07번 7b는 SCD "당시 기준" 조인이다. is_current=1 조인(현재 기준)과
  결과가 어떻게 다른지 비교해보면 SCD의 존재 이유가 눈에 보인다.
